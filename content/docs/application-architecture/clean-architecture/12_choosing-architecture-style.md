## 1. 어떤 아키텍처 스타일을 선택해야 할까

지금까지 헥사고날 아키텍처 스타일로 application을 구성하는 방법을 살펴봤다.

하지만 앞에서 다룬 질문의 답이 헥사고날 아키텍처에만 적용되는 것은 아니다.
계층형 아키텍처에도 적용할 수 있는 내용이 있고,
도메인 중심 접근에서만 의미 있는 내용도 있다.

경험상 좋은 방법이 아니라고 생각해 일부 내용에 동의하지 않을 수도 있다.
결국 가장 중요한 질문은 이것이다.

```text
헥사고날 아키텍처는 언제 사용해야 하는가?
```

---

## 2. 도메인이 왕이다

헥사고날 아키텍처의 핵심 목적은 도메인 로직을 외부 기술의 변경으로부터 보호하는 것이다.

```text
영속성 기술 변경 ─────┐
외부 시스템 변경 ─────┼──X──▶ Domain
UI / API 변경 ────────┘
```

도메인 로직은 JPA, SQL, HTTP, message broker 같은 기술적 관심사를 몰라야 한다.
외부 기술은 port와 adapter 뒤에 두고, 모든 의존성이 application core를 향하게 만든다.

그래서 헥사고날 아키텍처는 DDD(Domain-Driven Design)와 잘 어울린다.
도메인 중심 아키텍처는 DDD를 가능하게 만드는 조력자다.
도메인 모델이 외부 기술에 끌려가지 않도록 공간을 제공하기 때문이다.

따라서 첫 번째 판단 기준은 도메인 코드의 중요도다.

```text
도메인 규칙이 application의 핵심
  → 헥사고날 아키텍처의 효과가 큼

단순 CRUD와 데이터 전달이 application의 대부분
  → 헥사고날 아키텍처의 효과가 작을 수 있음
```

도메인 코드가 application에서 가장 중요한 부분이 아니라면,
헥사고날 아키텍처를 적용하기 위한 port, adapter, mapping 비용이 실질적인 이득보다 클 수 있다.

---

## 3. 경험이 여왕이다

많은 개발자는 계층형 아키텍처를 반복해서 사용해왔다.
그래서 계층형 구조가 자연스럽고 익숙하다.

익숙함이 반드시 나쁜 것은 아니다.
과거 프로젝트에서 편하게 개발하고 유지보수했다면,
그 경험은 계층형 아키텍처가 해당 환경에서 괜찮은 결정이었다는 증거다.

단지 새로운 스타일이라는 이유만으로 기존 방식을 버릴 필요는 없다.

아키텍처 스타일을 제대로 판단하는 가장 좋은 방법은 여러 스타일을 직접 경험하는 것이다.
책이나 설명만으로는 비용과 효과를 정확히 알기 어렵다.

헥사고날 아키텍처에 확신이 없다면 현재 application 전체에 한 번에 적용하지 않아도 된다.
작은 module이나 bounded context에 먼저 적용해볼 수 있다.

```text
application
├── 기존 계층형 module
├── 기존 계층형 module
└── 작은 헥사고날 module
    ├── adapter
    ├── application
    └── domain
```

실제로 적용하면서 다음을 확인한다.

1. 도메인 로직을 이해하고 변경하기 쉬워졌는가
2. 테스트하기 쉬워졌는가
3. port와 mapping code의 비용은 감당할 만한가
4. 팀원이 구조를 쉽게 이해하는가
5. 외부 기술 변경의 영향이 줄었는가

이 경험이 다음 아키텍처 결정을 이끌어준다.

---

## 4. 그때그때 다르다

어떤 아키텍처 스타일이 가장 좋은지 묻는다면,
저자나 전문 consultant도 대부분 "그때그때 다르다"라고 답할 것이다.

선택은 여러 조건에 따라 달라진다.

| 조건 | 확인할 질문 |
|---|---|
| software 성격 | 복잡한 비즈니스 application인가, 단순 CRUD인가 |
| 도메인 코드 역할 | 도메인 규칙이 핵심 경쟁력인가 |
| 변경 가능성 | DB, UI, 외부 시스템이 자주 바뀌는가 |
| 팀 경험 | 팀이 해당 스타일을 이해하고 유지보수할 수 있는가 |
| 비용 | port, mapping, module 분리 비용을 감당할 수 있는가 |

아키텍처 스타일은 목표가 아니라 수단이다.
헥사고날 아키텍처를 적용했다는 사실보다,
현재 software의 문제와 변경 방향에 맞는 구조를 선택했는지가 중요하다.

이 책은 하나의 정답 대신 판단에 도움이 될 원칙과 경험을 제공한다.
상황과 팀에 맞춰 적절한 아키텍처를 선택해야 한다.
